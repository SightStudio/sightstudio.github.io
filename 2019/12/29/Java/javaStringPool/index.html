1. String Pool

잘 알려진 사실이지만 자바의 String은 Immutable합니다.
자바에서 한번 assign된 문자열은 수정될 수 없으며, 문자열을 수정하면

기존의 문자열을 수정하는것이 아닌, 새로운 문자열을 만든다.

그렇기 때문에 자바에서는 동일한 문자열을 생성할때 드는 메모리, 생성 시간을 줄이기 위해
생성된 스트링 리터럴을 String Pool이라는 곳에 저장한다.
즉, String Pool에 있는 문자열이면 재사용하고, 없을 때만 새로운 문자열 객체를 만든다. 캐싱과 같은 개념이다.

  • 이러한 형태를 디자인 패턴에서는 Flyweight Pattern이라고도 한다.
  • String Pool의 위치는 JVM 구현체에 따라 다르다. eg. Hotspot VM, GrralVM
  • String이 아닌

하지만, 우리가 new 연산자를 사용하여 문자열을 만들 경우에는 String Pool에 저장되지 않는다.
반대로 String 클래스의 intern() 메소드를 사용하면, String Pool 밖에 있는 문자열 객체를
String Pool에 저장 할 수 있다.

2. 사진 설명

컴파일러는 컴파일시 변경의 여지가 없는 값은 미리 합쳐놓는다. (String 리터럴, final 상수)
사실 .equals() 만 쓰면 만날일 없는 문제… ( Object의 Equality를 비교하자 )
  1. “assert_” + “string” == “assert_string”
    -> 좌변을 컴파일러가 합쳐서 String Pool에 저장한다. = true
  1. new String(“assert_string”).intern() == “assert_string”
    -> 힙에 생성된 new String을 String Pool에 집어 넣는다. = true
  1. “assert_” + new String(“string”) == “assert_string”
    -> java 8 이하에서는 컴파일러가 StringBuilder로
    그리고 java 9 이후에는 StringConcatFactory.makeConcatWithConstants​로 변환한다.
    -> 모두 새로운 String 리턴 = true
  1. “assert_” + fString == “assert_string”
    -> 변경의 여지가 없으므로 컴파일러가 컴파일시 합쳐서 String Pool에 저장한다. = true
  1. “assert_” + vString == “assert_string”
    -> 3번과 동일하다. = false
  1. “assert_”.concat(“string”) == “assert_string”
    -> String.concat 내부에서 new String을 리턴한다. = false
